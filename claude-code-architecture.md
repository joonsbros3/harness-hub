## 개요

2026년 3월 31일, Anthropic의 Claude Code 소스코드가 npm 패키지에 `.map` 파일이 포함된 채 공개되었습니다. 이 글은 유출된 코드의 설계 철학과 아키텍처 인사이트를 정리합니다.

---

## 1. 지능은 모델에, 하네스는 얇게

Claude Code의 핵심 설계 원칙:

> "Intelligence belongs in the model, not the harness."

- 핵심 루프(TAOR: Think, Act, Observe, Repeat)는 약 50줄에 불과
- 계획 수립, 의사결정, 오류 복구는 모델이 직접 담당
- 모델 성능이 향상되면 하네스 코드를 삭제

> "Delete code when models improve."

하네스는 모델의 현재 한계를 보완하는 임시 비계이며, 모델이 성장하면 제거됩니다.

---

## 2. 4개 프리미티브로 모든 것을 합성

100개의 전용 통합 대신 4가지 기본 능력의 조합:

| 프리미티브 | 역할 |
|-----------|------|
| Read | 파일, 이미지 읽기 |
| Write/Edit | 정밀한 텍스트 조작 |
| Execute | 셸 명령 실행 |
| Connect | MCP 프로토콜로 외부 확장 |

**특징**:
- Edit이 가장 자주 호출되는 도구
- RAG 대신 LLM-native search 방식 채택
- ripgrep, jq, find 같은 유닉스 도구를 직접 활용

---

## 3. 메모리 아키텍처: 6계층 + 5단계 컴팩션

### 6계층 메모리 구조

| 계층 | 내용 | 특징 |
|------|------|------|
| 1 | 조직 관리 정책 | 제외 불가, 최우선 |
| 2 | 프로젝트 설정 (CLAUDE.md) | 팀 공유, VCS 커밋 |
| 3 | 사용자 개인 설정 | 전역 작업 스타일 |
| 4 | 세션 히스토리 | 현재 대화 맥락 |
| 5 | 자동 학습 패턴 | 200줄/25KB 캡 |
| 6 | 실시간 트랜스크립트 | 가장 먼저 삭제됨 |

**MEMORY.md**: 메모리 내용 자체가 아닌 포인터 목록으로 작동. Sonnet 모델이 턴마다 5개의 관련 메모리를 선택적으로 로드합니다.

### 5단계 컴팩션 전략

1. **Snip Compact** — 오래된 메시지 제거
2. **Microcompact** — 턴 내 도구 결과만 압축
3. **Auto-Compact** — 모델이 전체 요약 생성
4. **Reactive Compact** — API 에러 시 긴급 폴백
5. **Context Collapse** — 지능형 프루닝

우선순위 기반 정보 보존으로 조직 정책은 절대 삭제하지 않습니다.

## 4. 프롬프트가 아닌 구조로 강제

`bashSecurity.ts`는 2,592줄의 셸 명령 검증 모듈로 다중 계층 방어를 구현:

1. 정규식 패턴 탐지
2. Tree-sitter AST 분석
3. Heredoc 추출
4. 경량 모델의 2차 검증
5. 유니코드 경로 우회 방지

"rm -rf 쓰지 마"와 같은 프롬프트 기반 지침은 100% 신뢰할 수 없으므로 구조적 강제가 필수입니다.

---

## 5. 서브에이전트: 격리가 핵심

**설계 원칙**:
- 최대 1단계 깊이의 서브에이전트만 허용
- 서브에이전트의 재귀 생성 금지

**서브에이전트 격리 요소**:
- 독립된 컨텍스트 윈도우
- 별도 토큰 예산
- 제한된 도구 접근
- 격리된 MCP 연결

부모는 최종 요약만 받습니다. 자동 검증 서브에이전트가 별도로 트리거되어 자기 점검을 구조에 내장했습니다.

---

## 6. Skills와 Progressive Disclosure

**시간별 로딩 전략**:
- **세션 시작**: name과 description만 시스템 프롬프트에 로드
- **실제 호출**: 전체 내용 로드

수십 개의 스킬을 등록해도 컨텍스트 부담이 최소화됩니다. MCP 서버의 lazy loading과 동일한 철학입니다.

---

## 7. 모델 선택 전략

Claude Code는 중요 호출의 50% 이상에 Claude 3.5 Haiku를 사용:

- 파일 읽기, 웹 파싱, git 히스토리 처리
- 대화 요약
- 메모리 관련성 판단
- 보안 검증의 2차 판단

프리미엄 모델 대비 70~80% 비용 절감을 실현하는 경제적 설계입니다.

## 9. 프롬프트가 아니라 환경을 설계

**핵심 통찰**: "에이전트 성능의 병목은 모델이 아니라 구조다."

harness 프로젝트 실험 결과:
- 프롬프트만 사용: 49.5점
- 구조화된 harness 적용: 79.3점
- 모든 과제에서 우위, 분산 32% 감소

난이도별 효과:
- Basic: +23.8
- Advanced: +29.6
- Expert: +36.2

## 10. 주요 교훈

1. **얇은 오케스트레이션이 두꺼운 프레임워크를 이긴다**
2. **합성 가능한 프리미티브가 전용 통합보다 확장성이 우수하다**
3. **컨텍스트 관리에는 단일 전략이 통하지 않는다**
4. **안전은 프롬프트가 아닌 구조로 강제해야 한다**
5. **축소를 전제로 설계하라**
6. **진짜 경쟁력은 모델이 아닌 하네스 레이어에 있다**

코드 생성기의 다음 단계는 더 영리한 단일 에이전트가 아니라, 더 잘 설계된 환경과 구조입니다.
