# Harness-Hub 배경과 철학

---

## 1. 하네스 없이 바이브 코딩할 때 생기는 문제

- **방향 이탈** — 요구사항이 모호한 채로 시작하면 에이전트가 엉뚱한 방향으로 달린다. 한참 구현한 뒤에야 틀렸다는 걸 안다.
- **세션 리셋 비용** — 새 세션을 시작할 때마다 같은 컨텍스트를 다시 설명한다. 작업보다 설명이 더 오래 걸리는 상황이 생긴다.
- **범위 초과** — 지침 없이 요청하면 에이전트가 요청 범위 밖의 코드를 건드린다. 의도치 않은 사이드 이펙트가 생긴다.
- **도메인 지식 편차** — 같은 작업이라도 세션마다 판단 기준이 다르다. 일관된 코드 품질을 유지하기 어렵다.
- **검증 없는 PR** — 구현이 끝나면 바로 PR로 넘어가는 습관이 생긴다. 되감기 비용이 작업 비용보다 커지는 시점이 온다.
- **루프** — 위 문제들이 겹치면 에이전트가 같은 지점을 반복해서 건드리는 루프에 빠진다.

harness-hub는 이 문제들을 없애기 위한 게 아니라, **루프가 시작되는 지점에서만 잡아주는 최소한의 보조장치**다.

## 2. 핵심 철학

### "하네스는 얇게, 지능은 모델에"

2026년 3월 커뮤니티 소스 분석을 통해 Claude Code 설계 철학에 대한 인사이트가 공유됐다. 커뮤니티 해석 기반이지만, 실제 동작 방식과 높은 일관성을 보인다.

> "Intelligence belongs in the model, not the harness."
> "Delete code when models improve."

핵심 루프는 최소한으로 유지하고, 계획 수립·의사결정·오류 복구는 모델이 직접 담당한다는 것이 핵심이다. 하네스는 모델의 현재 한계를 보완하는 임시 비계이며, 모델이 성장하면 하네스 코드를 삭제한다.

이 원칙이 harness-hub의 기반이다. 규칙으로 모델을 통제하려는 게 아니라, 모델이 잘 동작할 수 있는 최소한의 구조만 제공한다.

**참조** (커뮤니티 소스 분석): [Claude Code의 구조 — 하네스는 얇게, 지능은 모델에](https://velog.io/@strurao/Claude-Code%EC%9D%98-%EA%B5%AC%EC%A1%B0-%EC%84%A4%EA%B3%BC-%ED%95%98%EB%84%A4%EC%8A%A4%EB%8A%94-%EC%96%87%EA%B2%8C-%EC%A7%80%EB%8A%A5%EC%9D%80-%EB%AA%A8%EB%8D%B8%EC%97%90)

### "빠르게 달리되, 특정 지점에서만 강하게 제동"

경험 많은 개발자에게 필요한 건 느리지만 안전한 프로세스가 아니다. 루프의 원인이 되는 지점 — 요구사항이 흐려질 때, 구현이 검증 없이 PR로 넘어갈 때, 컨텍스트가 새 세션에서 다시 처음부터 시작될 때 — 그 지점에서만 강하게 잡아주는 시스템이다.

나머지는 전부 권장 사항이다. 강제하지 않는다.

## 4. 구축 히스토리 (Wave 1~5)

처음에는 에이전트를 20개 가까이 만들었다. 역할이 세분화될수록 관리 복잡도가 폭발적으로 늘었다. 에이전트 간 책임 경계가 모호해지고, 어떤 에이전트를 써야 할지 매번 판단해야 했다. 이게 루프의 원인이 됐다.

Wave 1~5를 통해 구조를 정리했다.

| Wave | 변경 내용 | 이유 |
|------|---------|------|
| Wave 1 | pre-planner, plan-reviewer → planner 내부 통합 | 별도 에이전트로 관리할 가치 없음 |
| Wave 2 | 도메인 에이전트 8개 → skills/\*/SKILL.md의 Domain Expert Persona | 에이전트 수를 줄이면서 전문성 유지 |
| Wave 3 | analyzer, media-reader 제거 → Grep/Glob/Read 직접 사용 | 에이전트 위임 오버헤드 대비 이득 없음 |
| Wave 4 | delegator → orchestrator 흡수 (notepad 시스템 + 6-섹션 프롬프트) | 위임 레이어가 오히려 컨텍스트 손실을 유발 |
| Wave 5 | settings.json 권한 정리, 훅 경로 표준화 | 운영 일관성 확보 |

**결과**: 에이전트 20개 → 9개. 도메인 전문성은 skills 디렉토리로 이동.

## 5. 설계 결정들

### 왜 에이전트보다 스킬인가

에이전트를 추가하면 "어느 에이전트를 써야 하는가"라는 판단 비용이 생긴다. 스킬은 기존 에이전트(deep-worker)에 지식을 주입하는 방식이라, 판단 구조를 단순하게 유지하면서 도메인 전문성을 확보할 수 있다.

### 왜 notepad가 파일 기반인가

서브에이전트 간 상태 공유 방법으로 API나 메모리 시스템도 고려했다. 하지만 Claude Code가 파일을 읽고 쓰는 데 최적화되어 있고, 파일 기반이 디버깅도 쉽다. `.orchestrator/notepads/{계획명}/issues.md` 하나로 시작하는 것으로 충분하다.

### 왜 planner는 코드를 작성하지 않는가

계획과 실행을 같은 에이전트가 담당하면, 계획 단계에서 이미 특정 구현 방식으로 편향된다. planner는 인터뷰와 갭 분석에만 집중하고, 실행은 orchestrator가 담당한다. 역할 분리가 루프를 줄이는 핵심이다.

## 6. 어떻게 활용하는가

### 일상적인 흐름

```
평소 → 기본 세션 (claude)
복잡한 구현 → orchestrator (claude --agent orchestrator)
방향이 흐려지면 → planner로 전환
구현 후 → QA 게이트 → PR
컨텍스트 리셋 전 → /dev-docs-update
```

가장 중요한 두 가지:

1. **루프 조짐이 보이면 planner로 전환한다.** 계속 밀고 나가지 말고, 한 발 물러서서 요구사항을 정리한다.
2. **QA 없이 PR로 넘어가지 않는다.** 되감기 비용이 작업 비용보다 크다.

### 스킬을 의식적으로 쓰는 법

`skill-activation-prompt` 훅이 자동으로 제안하지만, 직접 로드할 때:

```
Skill("fe")          → React/Next.js 작업 전
Skill("be")          → API/DB 작업 전
Skill("commit-convention")  → 커밋 전
```

스킬은 "이 도메인에서 시니어처럼 판단하라"는 지시다. 로드하면 에이전트의 판단 기준이 바뀐다.

### 실패 패턴 축적

`.orchestrator/notepads/{계획명}/issues.md`는 단순한 메모다. 형식 없이 발견한 문제와 주의사항을 적는다. 이게 쌓이면 다음 루프를 막는 가장 실용적인 자산이 된다.

## 7. 개인화: 내 것으로 만드는 법

harness-hub는 범용 출발점이다. 실제로 쓰다 보면 내 스택, 내 작업 방식, 내 루프 패턴에 맞게 조정해야 한다.

### 스킬 추가

내가 주로 쓰는 스택이 harness-hub에 없다면 스킬을 추가한다.

```bash
mkdir ~/.claude/skills/my-stack
# SKILL.md 작성:
# - 핵심 원칙
# - 태스크-지식 매핑 테이블
# - ## Domain Expert Persona 섹션
```

`Skill("skill-developer")`를 로드하면 스킬 작성 방법을 안내받을 수 있다. 또는 `skill-creator` 플러그인을 사용한다.

### skill-rules.json 조정

자동 스킬 제안이 너무 자주 뜨거나, 필요한 스킬이 제안되지 않으면 `~/.claude/skills/skill-rules.json`의 키워드를 수정한다.

프로젝트마다 다른 규칙이 필요하면 `.claude/skills/skill-rules.json`을 프로젝트 루트에 추가한다. 글로벌보다 우선 적용된다.

### 에이전트 조정

agents/ 디렉토리의 `.md` 파일을 직접 편집한다. 각 파일은 해당 에이전트의 시스템 프롬프트다.

예: orchestrator가 특정 도메인을 자동 감지하게 하려면 `agents/orchestrator.md`에 도메인 트리거를 추가한다.

### 훅 조정

`settings.json`의 hooks 섹션에 이벤트별 셸 커맨드를 추가하거나 제거한다. 알림이 불필요하면 notification 훅을 제거. 린트 자동 실행이 필요하면 PostToolUse에 추가.

### CLAUDE.md 계층화

`/init-deep` 커맨드로 프로젝트별 CLAUDE.md를 자동 생성한다. 글로벌 규칙(이 레포의 CLAUDE.md)과 프로젝트별 규칙이 계층적으로 적용된다.

### 언제 하네스를 더 간소화할까

모델이 발전하면 일부 구조가 불필요해진다. Anthropic의 원칙대로, 더 이상 필요 없는 규칙은 삭제한다. "이 규칙이 실제로 루프를 막고 있는가?"라는 질문으로 주기적으로 정리한다.

## 8. 앞으로

harness-hub는 완성된 시스템이 아니다. 실제로 쓰면서 내 루프 패턴이 드러나고, 그걸 보완하는 방향으로 계속 진화한다.

다음 진화 방향과 현재 상태는 [FUTURE.md](FUTURE.md)에 정리한다.

> 에이전트 팀이 맞는 미래 방향이지만, 기능이 실험적이고 비용이 크다. solo harness가 먼저 검증된 뒤 도입한다.
